## Context

The interview flow is broken at multiple levels:
1. Frontend routing — setup page never renders under `/interview` (no child routes)
2. Backend persistence — evaluations created during `finishInterview()` are not saved to DB
3. Question loading — resume scenario has no question data loaded (stub method)
4. Auth verification — missing `/auth/status` endpoint breaks auth state after browser restart

These issues combine so that users see a blank page after login and cannot take interviews at all.

## Goals / Non-Goals

**Goals:**
- Users can see the interview setup page after logging in or registering
- Users can start an interview, answer questions, finish, and view results (including on reload)
- Resume of active sessions works — questions are loaded from backend
- Auth state survives browser restarts via `/auth/status` endpoint
- No duplicate/conflicting auth filters

**Non-Goals:**
- Refactoring the entire routing architecture (only add missing child routes)
- Changing the AI evaluation pipeline or response parsing
- Adding new features — only fixing broken ones
- Migrating to Spring Security's built-in JWT authentication

## Decisions

### Decision 1: Add child routes under `/interview` instead of using a single component with router-outlet

**Choice**: Create two child routes — `interview/setup` for the setup page and `interview/` as default (showing resume option or active interview)

```
/app/routes.ts changes:
{ path: 'interview', canActivate: [AuthGuard], children: [
    { path: '', loadComponent: InterviewPage },
    { path: 'setup', loadComponent: SetupPage }
]}
```

**Rationale**: The setup component is a separate page with its own form and flow. Using child routes gives us clean URL semantics (`/interview/setup`) and proper navigation state management. The router-outlet in InterviewPage renders the correct child based on the current route.

**Alternatives considered:**
- Render setup inline via ngIf — rejected because it makes the component bloated with both flows mixed together
- Use a separate top-level route like `/interview/setup` without parent-child relationship — rejected because then we'd need to duplicate auth guards and the resume check logic

### Decision 2: Persist evaluations in-memory before returning finishInterview response

**Choice**: In `InterviewController.finishInterview()`, save each Evaluation entity via `evaluationRepository.save()` immediately after parsing, before building the response.

```java
// Before building responses, persist evaluations:
for (Answer answer : answers) {
    Evaluation evaluation = ...; // parsed from AI response
    evaluationRepository.save(evaluation); // ← NEW: persist to DB
    
    // Build EvaluationDetail for response...
}
```

**Rationale**: The `Evaluation` entity already exists with all required fields. We just need to call save on it. No schema changes needed since the table already has the right columns (the entity is defined, just never persisted).

**Alternatives considered:**
- Create a separate `finishInterviewWithPersistence` method — rejected because it's unnecessary complexity for one extra call per evaluation
- Use `evaluationRepository.saveAll()` after building all evaluations — rejected because we need to build EvaluationDetail per-evaluation anyway, and save-per-iteration is cleaner

### Decision 3: Load questions from backend during resume via new endpoint or existing startInterview response pattern

**Choice**: Add a `/interview/session/{sessionId}/questions` GET endpoint that returns the session's persisted questions. The frontend calls this when resuming.

**Rationale**: Questions are generated by AI and stored in the `questions` table with a foreign key to `interview_sessions`. On resume, we need to reload them from DB since they're not kept in state across navigations. A dedicated endpoint is clean — it follows the existing REST pattern.

**Alternatives considered:**
- Return questions as part of `/interview/resume` response — rejected because that endpoint's purpose is specifically status, and adding questions mixes concerns
- Use startInterview with a resume flag — rejected because start creates new sessions, doesn't load existing ones
- Keep questions in the InterviewSignalService state — rejected because this is fragile (state lost on navigation/reload)

### Decision 4: Remove the custom anonymous filter from SecurityConfig, keep JwtAuthenticationFilter

**Choice**: Remove the inline anonymous `jakarta.servlet.Filter()` that sets request attributes. Move the attribute-setting logic into `JwtAuthenticationFilter`.

**Rationale**: Both filters are registered at the same position (`UsernamePasswordAuthenticationFilter.class`), which means their relative order is undefined. The custom filter may overwrite or be overwritten by JwtAuthenticationFilter's work, causing intermittent auth failures. By consolidating into a single well-defined filter, we eliminate this race condition.

**Alternatives considered:**
- Keep both but use distinct positions — rejected because the real problem isn't the position, it's that two filters do overlapping work and neither is aware of the other
- Use Spring Security's built-in JWT authentication — rejected because that would be a larger refactor (out of scope for this fix)

### Decision 5: Fix analytics queries to handle null categories gracefully

**Choice**: Add null-safe handling in `AnalyticsService` — check if category_id is null before casting, and use LEFT JOIN semantics properly.

**Rationale**: The native query groups by `q.category_id` but only joins CODE questions (`WHERE q.type = 'CODE'`). When a session has no CODE questions or the question has no category, `(String) row[2]` throws ClassCastException. Adding null checks prevents this crash.

## Risks / Trade-offs

[Risk: Evaluation save failure during finishInterview causes partial persist]
→ Mitigation: The existing try-catch in `finishInterview()` wraps each evaluation individually. If one fails, the session status is still set to COMPLETED and other evaluations are saved. We should add a transaction boundary so all-or-nothing persistence is possible.

[Risk: Adding child routes changes URL structure from `/interview` to `/interview/setup`]
→ Mitigation: The InterviewPage component still renders at both `/interview` (resume option) and `/interview/` (active interview). Only the setup page gets a new URL path. Users who bookmarked `/interview` will see the resume screen or redirect to `/login`.

[Risk: Removing custom filter may break auth for existing deployments that rely on request attributes]
→ Mitigation: The same logic exists in JwtAuthenticationFilter — it also sets `currentUserId` and `currentEmail` on the request. No functional change, just consolidation.

[Risk: Resume endpoint adds latency to interview page load]
→ Mitigation: Questions are typically small (a few KB), so the additional HTTP call is negligible. We can add caching later if needed.

## Open Questions
